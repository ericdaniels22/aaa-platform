## Decisions locked

Locked this session via explicit yes/proceed during execution:

- **C2 strategy for Task 11 type migration:** standalone deprecated `InvoiceWithItems` interface in `src/lib/invoices.ts` (not a re-export alias — shapes incompatible) for the three to-be-replaced legacy callers. Eric's refinements: JSDoc explicitly names the three files + tasks (14, 15, 47); includes the verification grep command (`grep -r "InvoiceWithItems" src/`); a one-line "Uses deprecated InvoiceWithItems — replaced in Task N" comment above the import in each Group C file; `tsc --noEmit` ran after migration to verify Group A's "no change needed" claim; plan doc Task 10 + Task 11 sections amended with notes documenting the deviation.
- **Plan-spec correction (`requirePermission` calling convention):** the spec'd `const denied = await requirePermission(...); if (denied) return denied;` doesn't tsc — the actual function returns `{ ok: true; userId } | { ok: false; response }`. Use the established pattern from `src/app/api/item-library/route.ts`: `const auth = await requirePermission(...); if (!auth.ok) return auth.response;`. Applied to Tasks 14 + 15. **Tasks 16–25 will need this same correction** — pass it into each subagent dispatch up front.
- **Plan-spec correction (`getActiveOrganizationId` nullability):** returns `Promise<string | null>` not `string`. When the orgId flows into a strict-string param (e.g., `createInvoice`'s `organizationId`), guard with `if (!orgId) return NextResponse.json({ error: "no active org" }, { status: 400 });`. When it flows into Supabase `.eq()` it's fine (matches existing `payments/route.ts` pattern).

**Lessons learned this session worth preserving:**

- **Type aliases preserve names, not shapes.** Task 10's initial commit aliased `InvoiceWithContents as InvoiceWithItems` thinking the rename would suffice — but `InvoiceWithContents` has `sections[]` (nested tree) and the old `InvoiceWithItems` had `line_items[]` (flat array). Six callers broke. Fix: standalone deprecated interface with the original flat shape. **Pattern to remember:** when migrating a type, grep for the old type name's *field accesses* (e.g., `.line_items.map`), not just imports. Names lie about shapes.
- **Plan specs have calling-convention drift.** The plan was written before the codebase shipped `permissions-api.ts` and `get-active-org.ts` in their current form. Always read the actual exported signatures from the lib before trusting a plan code block. The `requirePermission` correction surfaced clean during Task 14; if the plan had been spot-checked earlier this could have been caught at plan-review time.
- **For mechanical refactors (server libs Tasks 10/12/13), spec-only review is sufficient and fast.** Only Task 11 had judgment calls (scope of Group B rewrite, JSDoc wording). This confirms hybrid B is calibrated correctly — both reviewers on judgment-heavy tasks (2/8/28/44), spec-only on the rest.
- **Pre-67b invoices may render empty in the new sectioned PDF** — pre-67b line items have `section_id = NULL` so `getInvoiceWithContents` skips them. No real invoices currently exist (per 67b foundation handoff), so this is a transient state that doesn't matter. Worth noting if a real invoice appears in test data before Task 47 lands.
