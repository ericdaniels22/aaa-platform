## Decisions locked

- **Collapse-on-demand UX pattern.** Every major builder section (TotalsPanel, CustomerBlock, opening + closing statements, SectionCard, SubsectionCard) gets an inline chevron toggle. Default expanded. State is per-component `useState`, NOT persisted across reloads (matching the simplicity convention of other UI prefs in this app). If persistence is wanted later, localStorage is the path.
- **Back-link placement.** Edit page: HeaderBar far-left, icon-only on narrow viewports. Read-only view: breadcrumb-style "← Back to job" above the page header. Both use `next/link` for client-side nav.
- **`computeEstimateTotals` lives in `src/lib/estimates-calc.ts`** as pure functions that mirror the server's `recalculateTotals` math. The whole point is fidelity to the server, so `round2` is imported from `format.ts` (the server's source of truth) — never redefined locally.
- **One-shot operations stay immediate; field/line-item edits auto-save with 2s debounce.** Section/subsection create/rename/delete + line-item delete + line-item create (via AddItemDialog) all hit their endpoints immediately. Title / dates / statements / markup / discount / tax / status / individual line-item field edits are debounced through the auto-save hook. Reorder operations fire immediately on drag-end (not debounced — drag-end is an explicit user action).
- **`onAddSection` / `onAddSubsection` signatures extended from the plan.** Plan literal omitted the title argument; controller (and Eric implicitly via the SDD chain) approved adding it so the dialog logic stays inside the section card rather than the parent.
- **`section-card` / `subsection-card` interpret spec §5.1 as TWO separate sortable lists** per section — subsections in their own SortableContext, direct items in their own. No mixed sort_order space.
- **TotalsPanel uses `fixed bottom-right` (not `sticky`)** at desktop widths. Mobile responsiveness is post-67a polish; the panel will overlap on viewports narrower than ~768px (annotated with a comment in `estimate-builder.tsx`).

## Notes for next session

**67a SDD pipeline observations (lessons for future builds):**

1. **The "deferred Important findings" pattern worked well but cumulative debt is real.** Across Tasks 19, 21, 24, 26, 27, 28, the controller deferred 8+ Important findings into "follow-up cleanup" buckets. None were show-stoppers individually (the SDD reviewer always weighed in on whether to fix or defer), but together they form the 67a-routes hardening sweep chip + the I1/I2/I3 deferrals listed above. Recommend a dedicated 30-60 min session to clear them before kicking off 67b — tech debt compounds fast on a 30-task build.

2. **Auto-save (Task 28) had 2 Critical findings** that nearly slipped through: no fetch timeout (hung connections would stick saveStatus in "saving" forever) and no per-line-item in-flight guard (two debounced saves on the same item could race). The reviewer caught both and explicitly said "do not merge as-is." Both fixed before shipping. **Lesson: any task that introduces async network coordination deserves an extra-careful code review pass beyond the standard SDD chain.**

3. **The plan's literal API contracts didn't always match the codebase reality.** Examples: `createEstimate` lib helper named in the plan didn't exist (Task 19 inlined the logic instead). `onAddSection` / `onAddSubsection` signatures omitted the title argument (extended in Tasks 24/26). DropdownMenuTrigger `asChild` doesn't work with this codebase's base-ui (not Radix) variant. **Lesson confirmed: the plan/codebase pre-flight pattern is essential. Every Task prompt this session included a "Controller-Level Decisions" section that documented divergences before the implementer started.**

4. **The browser test was skipped throughout the session** because dev Supabase = prod Supabase per the worktree env. Eric's manual verification at the end caught one real bug (the jsonb damage_type filter) that none of the SDD reviews caught. **Lesson: the "skip browser test to avoid prod mutations" stance is correct, but means Eric's manual testing is the only safety net. Worth standing up scratch creds + scratch DB for 67b to enable browser tests in the SDD chain.**

5. **Real-prod-data side effect tracking.** Two test estimates were created + voided in AAA prod during today's verification activities. The handoff doc above lists them. Future sessions should explicitly track this side effect — it's easy to forget that "tested in browser" means "mutated production."

**UX polish was Eric-driven, not plan-driven.** The collapse pattern, back links, and TotalsPanel minimize were all Eric's requests after seeing the shipped 67a UI. Plan part-2 didn't anticipate them. This is healthy — manual interaction surfaces UX needs that specs alone miss. Recommend leaving 30-60 min budget after every UI-heavy build for "Eric pokes around the new feature, requests polish" iterations.

**For 67b kickoff:** the auto-save hook (`use-auto-save.ts`) is the most complex piece of the 67a codebase and will need extension for invoices (similar field shape, similar reorder semantics, additional fields like `invoice_number`, `payment_status`). Plan 67b should explicitly say whether the hook gets generalized or duplicated. The current hook is hard-coded to `EstimateWithContents` and the `/api/estimates/...` URL prefixes; either generic-ify it (with a config object passed in) or fork into a parallel `use-invoice-auto-save.ts`.
