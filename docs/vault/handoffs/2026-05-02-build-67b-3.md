**Two patterns worth remembering for future builders:**

1. **Pre-flight plan code blocks against the actual codebase before dispatching.** This session caught 5 divergences in the first 5 minutes (the `<EstimateBuilder mode=...>` API, missing `<Combobox>` UI primitive, outdated `requirePermission`/`getActiveOrganizationId` calling conventions, missing `router.refresh()` after apply-template, plan-assumed nested settings-nav vs actual flat one). Each divergence would have caused a fix iteration during dispatch; pre-flight collapsed them into a single Task 33.5 + plan inline-edit pass before any subagent ran. **Pattern is now established: 5-minute pre-flight on plan code blocks before kicking off any new milestone batch.**

2. **`useState(prop)` initializes once.** When a parent re-fetches and re-renders with a new prop value, the child's local-state copy ignores the update unless explicitly re-synced. The fix is a `useEffect` keyed on a stable identifier (here: `[entity.data.id, entity.data.updated_at]`). The `eslint-disable-next-line react-hooks/exhaustive-deps` was needed because the linter wants the entire `entity` object in deps, but that would clobber every parent re-render and overwrite optimistic auto-save state. Surfaced during Test 2 (apply-template UI didn't update). **Worth scanning for other `useState(prop)` patterns elsewhere in the builder shell** ‚Äî anywhere local state mirrors a server-fetched prop is suspect.
