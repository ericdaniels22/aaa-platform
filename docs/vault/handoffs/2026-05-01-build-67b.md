## Decisions locked

Locked this session via explicit yes/proceed during brainstorm and execution:

- **67b scope:** full canonical (templates + invoice editor + conversion) per build guide §7.
- **Builder shell pattern:** `mode: 'estimate' | 'invoice' | 'template'` prop on existing 11 builder components, prop-drilled (no context provider). Shipped baseline in Task 8.
- **Auto-save hook:** generic-ified with `AutoSaveConfig<T>` config object. Templates skip snapshot-409. Shipped in Task 7.
- **Existing invoice UI fate:** truncate old data is N/A; replace create/edit UI wholesale; **keep read-only detail + payment surface** (Stripe payment-request modal, record-payment modal, `mark-sent`/`pdf`/`send`/`void` routes) untouched.
- **Apply Template UX:** post-create banner that disappears after first apply (localStorage flag) OR sections.length > 0. Broken-refs warning surfaces as toast + dismissible persistent banner.
- **Conversion UX:** plain confirm modal listing what happens in bullets; no editable fields.
- **Send semantics:** "Mark as Sent" status flip only in 67b; 67c will add a separate "Send via Email" button that does PDF + email.
- **Rejection state:** terminal (no back-to-draft path). Customer-facing approval magic links deferred to 67c+.
- **Templates nav placement:** under existing Catalog group, next to Item Library.
- **SDD execution mode:** hybrid B — both reviewers (spec compliance + code quality) on Tasks 2/8/28/44; spec-only review on the other 48.
- **Branch strategy:** direct-on-main, matches 67a precedent.
- **Spec deviation (Part 2 preamble):** template route table collapsed from spec §7's 7 routes to 2. Granular section/line-item routes are NOT built for templates. Template-mode auto-save uses rootPut-only with full-structure JSONB body. Sub-100ms full-structure JSONB writes for typical 5–20 line-item templates. If templates grow huge later, granular routes can be added without breaking the shell.
