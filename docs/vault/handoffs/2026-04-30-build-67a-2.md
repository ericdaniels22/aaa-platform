## Notes for next session

**SDD pacing observation.** Five tasks via SDD with two-stage review
took roughly 60 minutes of wall clock. That's ~12 minutes per task
including implementer + spec reviewer + code-quality reviewer + (in
one case) C1 fix + re-review. Tasks 10–14 should clock similarly —
plan accordingly when scoping the next session.

**Hardening template lives in part-2's "Hardening notes" section.**
Tasks 10–14 already incorporate every pattern the reviewer flagged in
Tasks 7–9: JSON parse try/catch, ID type-check, length caps, null-org
guard, 404-before-mutate, soft-delete idempotency. The implementer
subagent should NOT reduce these to match Tasks 7–9's minimalism —
they're the new standard from Task 10 onward, and the deferred sweep
backports them to Tasks 7–9.

**The C1 lesson generalizes.** PostgREST's `.is()` filter is for
SQL `IS NULL/TRUE/FALSE/UNKNOWN` only. Any future query that does
`.is("col", value_that_might_be_a_uuid_or_null)` is broken. The
canonical pattern is the branch in `4e53cc3` — `.eq` for the value
case, `.is(_, null)` for the null case. If/when more such queries
get written, this is the gotcha.

**Code review verdict pattern.** The code-quality reviewer's
"Verdict: ship it (with follow-ups for X, Y, Z)" was the dominant
shape across Tasks 5, 6, 7, 8 — small finite list of "important
before this hits real traffic" items, none Critical. Task 9 was the
exception (C1, real bug, fixed mid-session). Future SDD chains should
expect this distribution: most reviewer findings become follow-ups,
not in-loop fixes; only Critical/wrong-on-purpose findings should
trigger a re-implement.

**`requirePermission` doesn't natively OR multiple keys.** Tasks
13, 14 (item-library API routes) need GET to allow either
`view_estimates` OR `view_invoices`. The plan ships an inline
`requireEither` helper; promoted to `src/lib/permissions-api.ts`
later if more routes need OR semantics.

**Tasks 15–30 each reference a section of the spec doc and one or
two existing files for reference patterns.** The implementer subagent
should read those before writing — don't try to invent new patterns.
The form-builder code (`src/components/form-builder/canvas.tsx` for
dnd-kit, `use-form-config.ts` for auto-save) is the canonical
reference.
