## Recently learned

- **For builder-shell prop type widening across modes, use a discriminated union over the props object — not a single broadened `entity` prop.** Task 28's HeaderBar refactor: prop type went from `{ estimate: Estimate; ... }` to `{ mode: "estimate"; entity: Estimate; ...common } | { mode: "invoice"; entity: Invoice; ...common } | { mode: "template"; entity: TemplateWithContents; ...common }`. This lets tsc narrow `entity` cleanly inside each mode-branch without casts (`entity.status` resolves to the entity-specific status enum). A single broadened `entity: Estimate | Invoice | TemplateWithContents` prop with `mode` as a separate discriminator does NOT auto-narrow without an explicit type predicate or `if (mode === "X") { const e = entity as X; ... }` cast. **Pattern to remember:** when threading a discriminator and a polymorphic value through a component, lift them into the union at the props level so tsc's narrowing is automatic. Also useful when this same pattern lands on Task 41/42's template builder host and Task 47's read-only invoice view.

- **Plan code blocks lie about which files fetch in this codebase.** Task 32 plan listed 4 files (section-card, subsection-card, line-item-row, add-item-dialog) needing entityBase URL rewriting. In actuality: only add-item-dialog.tsx makes its own fetches; the 3 layout components are pure presentational and delegate every mutation through parent callbacks. Caught by a 30-second `grep -nE "fetch\(.*api/estimates"` before dispatching. **Pattern:** for any plan task that lists multiple files with the words "all", "each", or "every", grep first; the plan was drafted against a hypothetical codebase shape that may not match. Especially relevant for Tasks 41/42 (template editor host) and 47 (invoice read-only rewrite) where the plan likely assumes patterns from earlier prototypes.

- **Type aliases preserve names but not shapes.** During 67b Task 10, the initial commit aliased `InvoiceWithContents as InvoiceWithItems` thinking the rename would suffice for the kept callers — but the new type has nested `sections[]` and the old type had flat `line_items[]`. Six callers broke at tsc (PDF route writes `invoice.line_items = items`, PDF doc reads `invoice.line_items.map`, etc.). Fix at `9eae494`: standalone deprecated `InvoiceWithItems` interface holding the original flat shape, scoped to legacy callers being replaced in Tasks 14/15/47. **Pattern:** when migrating a type, grep for the old type name's *field accesses* (`.line_items.map`, etc.), not just imports. Names lie about shapes. The C2-strategy interview with the user (Option 1 / 2 / 3 trade-off) plus structured JSDoc + verification grep command is the canonical way to handle this.

- **Plan code blocks drift from actual codebase signatures.** The 67b plan's spec'd `requirePermission` calling convention (`if (denied) return denied;`) doesn't tsc — the actual function returns a discriminated union (`{ ok: true; userId } | { ok: false; response }`). Same with `getActiveOrganizationId` returning `string | null` despite plan code passing it directly into a strict-string param. Surfaced in 67b Task 14, applied uniformly going forward. **Pattern:** before trusting a plan code block's import + call, read the actual exported signature from the lib. A 5-minute pre-flight against `permissions-api.ts`, `get-active-org.ts`, `api-errors.ts`, `postgrest.ts` would have caught both before any subagent dispatched. Worth doing once at plan-review time rather than ten times during execution.

- **PostgREST `.is(col, value)` only accepts `null`/`true`/`false`/`unknown` — passing a UUID silently fails at runtime.** Surfaced during [[build-67a]] Task 9 code review (commit `0aae288`); fixed at `4e53cc3` by branching the query: `.eq` for the UUID case, `.is(_, null)` for the null case. The TS type system signs off because `supabase-js` types `.is()`'s second arg loosely. Any future query that does `.is("col", value_that_might_be_a_uuid_or_null)` against a nullable foreign-key column has the same bug. The canonical pattern is: `if (uuid) q = q.eq(col, uuid); else q = q.is(col, null);`. Plan part-2 calls this out in the "Hardening notes" section.

Both were collapsed to match the existing pattern (Eric chose Option A for Task 15 explicitly; controller adjusted Task 18 with announcement). **Pre-flight the references named in plan part-2 against the actual codebase before locking SDD prompts for Tasks 19–30.** Especially: confirm `src/components/form-builder/canvas.tsx` has the dnd-kit exports the builder UI tasks expect, and `src/components/tiptap-editor.tsx` has the Tiptap wrapper the statement editors expect.

- **`form_config` had a long-standing latent bug from build46.** Migration build46 added a unique index `form_config_org_key` on `(organization_id)` alone — collapsing each tenant to a single row. The unchanged `/api/settings/intake-form` POST route did `INSERT` with `version+1`, so every save after the first per-org silently 500'd with a duplicate-key violation. Eric had been editing AAA's form for months without noticing — the toast read "Save failed" and was assumed transient. Surfaced during [[build-14j]] verification when the new always-visible version pill made the failure mode loud. Fix: [[build-14j]] Task 0 migration replaced the singleton with a composite `(organization_id, version)` unique. **Worth scanning other settings pages for the same shape** — any POST route that does `INSERT` with `version+1` against a table whose unique index is `(organization_id)` instead of `(organization_id, version)` will have the same issue.

- **PostgREST `.contains(col, jsArray)` against a `jsonb` column emits invalid Postgres syntax.** supabase-js's `.contains()` serializes a JS array as a Postgres array literal `{x,y}`. That works against `text[]` columns but fails against `jsonb` columns with `"invalid input syntax for type json"` because PostgREST's `cs` operator on jsonb requires the value to be JSON-formatted (`["x","y"]`). Surfaced during [[build-67a]] post-ship browser verification — `damage_type_tags` and `section_tags` are `jsonb NOT NULL DEFAULT '[]'::jsonb` (per migration-build67a, lines 24-25), and AddItemDialog's library tab toasted the error when Eric clicked into the dialog with a damage_type chip pre-applied. Fix at `3bc2617`: pass JSON-stringified value — `.contains("damage_type_tags", JSON.stringify([value]))`. **Canonical pattern:** if the column is jsonb (check the migration), wrap the array in `JSON.stringify()`; if text[], pass the JS array directly. **Worth a sweep across the codebase** — `grep -r "\.contains(" src/` to find other callers and verify each against its column type.

- **Auto-save fetches need explicit timeouts.** A hung connection (mobile network drop, slow server) without `AbortController` makes React's saveStatus stick in `"saving"` forever; `inFlightRef` stays `true`, blocking subsequent debounced saves. The estimate builder's `useAutoSave` hook hit this in code review (commit `cb26a0b` fix) — added `FETCH_TIMEOUT_MS = 30_000` + `AbortController` to all four save paths (estimate-level PUT, line-item PUT, sections-reorder PUT, line-items-reorder PUT). AbortError is routed to the same error path as network errors. **General lesson:** any client-side fetch that sets a "saving" status flag needs a timeout-and-abort cycle; otherwise the flag is a lie under network failure.

- **Snapshot-based concurrency guards must bump the parent's `updated_at` even on no-op-looking child mutations.** The 67a I2 fix (commit `4a85c7e`) added server-side `updated_at_snapshot` enforcement to the sections + line-items reorder PUTs. Reorders only touch child rows (`estimate_sections.sort_order`, `estimate_line_items.sort_order` / `section_id`); the parent `estimates` row would otherwise stay quiet, so two concurrent reorders against the same snapshot would both succeed and the second would silently overwrite the first's order. Fix: every mutation route that the snapshot guard protects must also `touchEstimate(supabase, estimateId)` (no-op `UPDATE estimates SET updated_at = now()` — the BEFORE-UPDATE trigger handles the actual value) so the next snapshot read sees the change. New helpers in `lib/estimates.ts`: `checkSnapshot` + `touchEstimate`. **General lesson:** if a route is guarded by `updated_at_snapshot` against a parent table but only writes to a child table, you must explicitly bump the parent's `updated_at` after the child write or the guard is a lie under concurrency.

- **PostgREST `.or()` accepts user input as comma-separated filter syntax — escape it.** The 67a Phase C sweep fixed three callers (`lib/item-library.ts`, `components/job-detail.tsx`, `api/email/contacts/route.ts`) where user-typed input was concatenated into an `.or(...)` filter expression. A search like `foo,bar` would be parsed as two filters with `bar` as a column-name fragment, which fails the request; worse, a value like `,deleted=is.true` could in principle escape into additional conditions (RLS contains the blast radius but the bug is real). Fix: new `lib/postgrest.ts` → `escapeOrFilterValue` wraps the value in PostgREST's documented quoted form (`"value"`) and escapes embedded `\` / `"`. `%` and `_` stay as ilike wildcards by design. **Canonical pattern:** any `.or(...)` or `.and(...)` whose value contains user input goes through the helper.

- **API routes should never return raw `error.message` on 5xx.** Postgres / Supabase errors leak schema details, constraint names, RLS policy text, and sometimes the offending row contents. Phase C added `lib/api-errors.ts` (`apiError` + `apiDbError`) that logs the full message server-side and returns `{error: "internal error"}` to the client. Wired into all 67a + item-library routes; the remaining ~80 routes still leak and are filed as a follow-up sweep. **Canonical pattern:** at every 5xx site, replace `NextResponse.json({ error: error.message }, { status: 500 })` with `apiDbError(error.message, "<METHOD> <route> <op>")`.
